# 构建脚本 bug 修复总结

## 问题分析

通过分析错误日志，我们发现了两个关键问题：

1. **EasyX 下载链接 404 错误**
   - 原始链接 `https://easyx.cn/download/EasyX_20241018.exe` 返回 404 错误
   - 这导致构建过程无法自动安装 EasyX 图形库

2. **Unicode 编码问题**
   - Windows 命令提示符默认使用 GBK 编码
   - 脚本中的 emoji 字符（如 ❌、💡）无法在 GBK 编码下显示
   - 导致日志输出时出现 `UnicodeEncodeError`

## 修复方案

### 1. EasyX 下载链接修复

**实现了 `get_latest_easyx_url()` 函数**，具有以下功能：

- **多页面尝试**：尝试访问多个可能的下载页面 URL
- **链接过滤**：过滤掉包含非 ASCII 字符的链接（如 "春分版"）
- **链接验证**：测试每个链接是否返回 200 状态码
- **备用链接**：提供多个版本的备用链接，从最新到旧版本
- **容错机制**：当所有链接都失败时，返回一个最可能的链接

**成功找到的有效链接**：`https://easyx.cn/download/EasyX_20220901.exe`

### 2. Unicode 编码问题修复

**实现了 `setup_logging()` 函数**，具有以下功能：

- **修复 stdout/stderr 编码**：将输出流编码设置为 UTF-8
- **日志文件编码**：确保日志文件使用 UTF-8 编码
- **移除 emoji**：用文本-based 指示器替代 emoji（如 `[OK]`、`[ERROR]`、`[WARN]`、`[SKIP]`）
- **修复日志配置**：移除重复的日志配置，确保只在 `setup_logging()` 中配置一次

## 如何使用

1. **以管理员身份运行**：脚本需要管理员权限来安装依赖
2. **默认自动模式**：脚本会自动尝试安装所有依赖
3. **命令行选项**：
   - `--install-mode=auto`：自动安装（默认）
   - `--install-mode=winget`：使用 winget 安装
   - `--install-mode=manual`：仅检查，不自动安装
   - `--skip-easyx`：跳过 EasyX 安装检查
   - `--skip-cmake`：跳过 CMake 安装检查
   - `--build-only`：只构建，不运行
   - `--clean`：清理旧构建文件
   - `--verbose`：显示详细输出
